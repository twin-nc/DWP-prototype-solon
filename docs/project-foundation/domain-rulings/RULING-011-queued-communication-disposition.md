## Rule Statement

### 1. Statutory suppression: DEBT_COLLECTION communications must be discarded

Communications with `communication_category = DEBT_COLLECTION` that were queued during a **statutory suppression period** must be **discarded** on suppression lift — they must not be dispatched.

Statutory suppression reasons:
- `BREATHING_SPACE_STATUTORY`
- `MENTAL_HEALTH_CRISIS_STATUTORY`
- `INSOLVENCY_STATUTORY`
- `DECEASED_MANDATORY`

**Rationale:** These communications are stale. The circumstances that generated them may no longer be valid (debt may have been paid, arrangements may have changed, vulnerability may have been identified). A fresh communication must be generated by the resuming process if still required.

### 2. Internal policy suppression: DEBT_COLLECTION communications must be held for agent review

Communications with `communication_category = DEBT_COLLECTION` queued during an **internal policy suppression** must be **held for agent review** on suppression lift. They must not be auto-dispatched, but they are not discarded — the debt relationship is unchanged and the communication may still be appropriate.

Internal policy suppression reasons:
- `DISPUTE_INTERNAL`
- `VULNERABILITY_POLICY`

**On suppression lift:** The system must create an agent task: "Review held communications for [account] — confirm appropriateness before dispatch." The agent must explicitly approve or discard each held communication before it is sent.

### 3. NON_COLLECTION communications must never be suppressed by statutory reasons

Communications with `communication_category = NON_COLLECTION` must not be blocked by statutory suppression reasons. They must not enter the queue in the first place if suppression is active. `CommunicationSuppressionService.isPermitted()` must permit `NON_COLLECTION` communications regardless of suppression state.

**Exception: DECEASED_MANDATORY** — see section 4.

### 4. DECEASED_MANDATORY suppression: all communications to the deceased are blocked

`DECEASED_MANDATORY` suppression blocks **all** communications — including `NON_COLLECTION` — addressed to the deceased person. The deceased person's preferred channel must not be used for any communication.

**Exception:** Communications to an executor or administrator of the estate are permitted once executor authority has been verified. These use a new communication category: `ESTATE_ADMINISTRATION`. This category is not subject to `DECEASED_MANDATORY` suppression once authority is verified.

---

## Edge Cases

### Communication was queued during a period that spanned both statutory and internal suppression

If a communication was queued during a period where both `BREATHING_SPACE_STATUTORY` and `DISPUTE_INTERNAL` were active, and both are now lifted, the statutory rule governs: the communication must be discarded (not held for review). The statutory discard rule takes precedence over the internal hold-for-review rule.

### Multiple communications queued for the same template/trigger

If the same template has been queued multiple times during a suppression period (e.g., monthly statement), only one should be regenerated on suppression lift — the most recent applicable one. The system must deduplicate before generating fresh communications.

### Suppression lifted and re-activated quickly

If suppression is lifted and a new suppression is activated within a short window (e.g., breathing space ends and a dispute is immediately raised), communications that were discarded on first lift must not be re-queued. Each communication lifecycle begins fresh.

## Data Classification Flags

No new Restricted or special-category-adjacent elements introduced by this ruling. `communication_category` and `suppression_reason` are Operational.

---

## Guardrails — Builders Must Not Violate

1. The `CommunicationDispatchService` must check the suppression reason that caused a communication to be queued before dispatching it on suppression lift. It must not simply dispatch all queued items when any suppression is lifted.
2. Communications discarded under rule 1 must have their status set to `DISCARDED_STATUTORY` in the communication log. They must not be deleted — the audit trail must show they existed and were discarded.
3. Communications held under rule 2 must have their status set to `HELD_FOR_REVIEW` and must generate an agent task. They must not be dispatched without explicit agent approval.
4. `CommunicationSuppressionService.isPermitted()` must receive `communication_category` as a parameter and apply the category-specific rules above. A single boolean `is_suppressed` flag is insufficient.
