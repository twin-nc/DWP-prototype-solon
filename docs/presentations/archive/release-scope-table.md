# SUM-04: Release Scope Table

Release 1 means working software in a running local/dev environment with the core architecture, high-value journeys, and evidence paths demonstrable. It is not a feature-complete DCMS, and policy-gated or still-undesigned areas remain explicit exclusions until their design inputs are closed.

| Scope area | Release 1 | Release 2 | Release 3 |
|---|---|---|---|
| Product goal | Prove the end-to-end DCMS baseline: case intake, workflow, role-scoped staff UI, audit, suppression, repayment, and strategy-governance slices running together. | Fill the policy-gated operational domains and complete the missing scaffolds for broader delivery coverage. | Complete the remaining whole-system lifecycle, correction, legal, finance, migration, and recovery capabilities for a production-shaped platform. |
| Modules in | `customer`, `account`, `strategy`, `workallocation`, `repaymentplan`, `payment`, `communications`, `integration`, `user`, `audit`, `analytics`, `reporting`, frontend shell/workspaces, infrastructure, CI/CD, docs governance. | Add or complete `thirdpartymanagement`, `infrastructure/process`, `shared/process/port`, DCA APIs, richer analytics, MI projection migrations, and self-service API hardening. | Add complete migration tooling, finance operations depth, legal/enforcement workflows, complaints, retention/archive, manual amendments, and operational recovery workbench. |
| Demo flows covered | Primary coverage for Flows 1, 2, 5, 6, 10, 12, and 13; partial supporting coverage for Flows 3, 7, 8, 9, 11, and 14 where the foundation modules are already present. | Complete Flows 3, 7, 8, 9, 11, and 14 by adding DCA, self-service handoff, dispute, deceased, guided-agent, and settlement depth. | Add production-scale variants of all flows, with migration, retention, correction, legal, complaint, and recovery edge cases. |
| Explicit exclusions | Not feature-complete; no public customer portal UI; no final DWP policy values for I&E, DCA notice periods, write-off thresholds, statute-barred timing, or champion/challenger gates; no complete operational recovery workbench. | Still excludes full legal case management if not brought into scope, full legacy migration execution, and complete retention/archive lifecycle unless specifically prioritised. | No strategic exclusions expected; remaining scope depends on DWP sign-off, production environment definitions, and release evidence gates. |
| Design dependencies | Master design, ADRs, RBAC decisions, communications/work-allocation/reporting packs, MI UX design, self-service integration baseline. | Policy closure for Tier B blockers; DCA, I&E, vulnerability, communications fallback, work-allocation SLA, and champion/challenger thresholds. | New or completed design packs for Tier A gaps: complaints, legal/enforcement, finance operations, amendments, retention/archive, migration, and recovery tooling. |
| Definition of done | Running local/dev stack, role-based login, core APIs and screens, auditable journey evidence, traceable design links, and demo-ready flow evidence. | Wider operational flow completion, pending-scaffold modules created, policy-gated behaviours implemented after sign-off, and broader integration evidence. | Production-shaped operational readiness with migration, retention, correction, recovery, legal/complaints, release evidence, and environment-readiness evidence complete. |
