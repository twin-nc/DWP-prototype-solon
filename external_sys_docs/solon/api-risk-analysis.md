# Risk Analysis API — Reference
**Version:** 2.1.0  
**Base path:** `/risk-analysis` (assumed; routed via Solon API Gateway)  
**Auth:** Bearer JWT from Keycloak on every request

> Solon's risk analysis service manages candidate lists and risk scoring for debt collection targeting. It supports Champion/Challenger workflows by grouping party roles into candidate lists, scoring them, and initiating debt collection interventions (case creation) for selected candidates. Operational Plans can be linked to candidate lists for campaign-level tracking.

---

## Endpoints

### Candidate List Types (Configuration)

| Method | Path | Summary |
|---|---|---|
| GET | `/candidate-list-types` | Search types (filters: `code`, `description`, `active`, `caseTypes`) |
| POST | `/candidate-list-types` | Create a new type |
| GET | `/candidate-list-types/{code}` | Get by code |
| PUT | `/candidate-list-types/{code}` | Update |
| DELETE | `/candidate-list-types/{code}` | Delete |
| GET | `/candidate-list-types/{code}/exists` | Check existence (returns boolean) |
| GET | `/candidate-list-types/dropdown` | Dropdown list (code + label) |

**`CandidateListType` schema:**

| Field | Type | Notes |
|---|---|---|
| `code` | string (max 10) | Unique identifier |
| `description` | string (max 50) | Human-readable label |
| `active` | boolean | Whether this type is in use |
| `caseTypes` | array | Allowed case types for candidates of this list type |
| `comment` | string | Configuration note |

**Validation:**
- `active`, `code`, `description` are required
- `caseTypes` filters which debt case types can be initiated from candidates in this list

| Method | Path | Summary |
|---|---|---|
| GET | `/candidate-lists` | Search lists (filters: `id`, `description`, `candidateListTypeCode`, `operationalPlanId`) |

**`CandidateList` schema:**

| Field | Type | Notes |
|---|---|---|
| `id` | string | **Null on CREATE**, non-null on UPDATE |
| `candidateListTypeCode` | string | **Required** — references a configured CandidateListType |
| `description` | string (max 50) | **Required** — list description |
| `candidates` | array | List of Candidate objects |
| `operationalPlan` | object | Optional link to an OperationalPlan (`id`, `description`) |

**Validation:**
- `id` must be null on CREATE, non-null on UPDATE
- `candidateListTypeCode`, `description`, and `id` are required

### Candidates (Nested in CandidateList)

Candidates are the individual party roles (debtors) within a candidate list.

**`Candidate` schema:**

| Field | Type | Notes |
|---|---|---|
| `candidateId` | string | **Null on CREATE** |
| `candidateListId` | string (max 36) | Null when creating from candidateList endpoint; non-null on manual create |
| `candidateSourceCL` | string | **Required** — from `CANDIDATE_SOURCES` — how this candidate was identified |
| `candidateStatusCL` | string | From `CANDIDATE_STATUS` — lifecycle state of the candidate |
| `partyRole` | Identification | **Required** — the party role (debtor) this candidate represents (UUID 36 chars) |
| `caseTypeCode` | string (max 10) | The debt case type to create on intervention initiation |
| `caseIdentification` | Identification | The case linked to this candidate (once intervention is initiated) |
| `riskScore` | number (0–9999.9999) | Risk score computed by the risk engine |
| `operationalPlan` | object | Optional link to an OperationalPlan |
| `dataArea` | string | Free JSON blob — custom metadata |

**`Identification` sub-schema:**

| Field | Notes |
|---|---|
| `id` | UUID (exactly 36 chars) — the entity's ID |
| `identificationLabel` | Read-only label generated by Solon |

### Intervention Initiation

| Method | Path | Summary |
|---|---|---|
| POST | `/candidates/intervention-initiation` | Initiate debt collection cases for selected candidates |

**`CandidateInterventionInitiationRequest` schema:**

| Field | Type | Notes |
|---|---|---|
| `candidatesIds` | array of string | IDs of candidates to initiate intervention for |
| `caseTypeCode` | string | **Required** — the debt case type to create |
| `operationalPlanId` | string | Optional — links the resulting cases to an operational plan |

**Validation:**
- `caseTypeCode` must reference an existing Case Type Code
- `operationalPlanId` must point to a valid operational plan if present

> This is the trigger that converts scoring candidates into active debt collection cases — calling `POST /case-management/cases` internally for each selected candidate.

### Risk Screening

Risk screening is triggered by Solon's BPMN engine internally via the `irm.bpmn-engine.screen-risk` Kafka command (private channel — not publishable by the additional layer). Results are returned via RISK_RESULT_SIGNAL.

The additional layer reads screening results as part of case/candidate data.

**`RiskScreening` schema:**

| Field | Type | Notes |
|---|---|---|
| `screeningStatus` | string | Status of the screening result |
| `overallRiskScore` | integer | Total risk score across all dimensions |
| `riskDimensions` | array | Breakdown by dimension |
| `errorMessage` | string | Error message if screening failed |

**`RiskDimension` schema:**

| Field | Notes |
|---|---|
| `description` | Description of this risk dimension (e.g. payment history, account age) |
| `score` | Total score for this dimension |
| `riskObservationRules` | Array of RiskObservationRule — individual rule contributions |

**`RiskObservationRule` schema:**

| Field | Notes |
|---|---|
| `description` | Description of the rule that fired |
| `score` | Score contribution from this rule |

## Architectural significance for the additional layer

### Champion/Challenger targeting

The additional layer's **Champion/Challenger Framework** uses candidate lists to group and score accounts for different collection strategies:

1. Risk scoring engine (Drools) produces `riskScore` values for candidates
2. `POST /risk-analysis/candidate-lists` — creates a candidate list linked to an operational plan
3. Candidates with higher risk scores are assigned to champion strategies; lower-risk to challenger strategies
4. `POST /risk-analysis/candidates/intervention-initiation` — triggers case creation for selected candidates

### Operational Plan linkage
`CandidateList.operationalPlan` and `Candidate.operationalPlan` confirm that candidate lists feed into operational plan tracking — the Supervisor Dashboard's campaign completion metrics are driven by cases initiated from these candidate lists.

### Risk screening boundary
The additional layer does **not** trigger risk screening directly — this is handled by the BPMN engine's internal `irm.bpmn-engine.screen-risk` command. The additional layer reads the `riskScore` on candidates after screening is complete.

### Strategy Engine integration
The Strategy Engine reads `GET /risk-analysis/candidate-lists?operationalPlanId={id}` to display candidate list population for a campaign. The Split Assignment Service uses `candidateSourceCL` and `candidateStatusCL` to track which candidates have been processed.
